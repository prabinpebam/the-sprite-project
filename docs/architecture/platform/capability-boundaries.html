Architecture · Platform contract

Capability Boundaries

The baseline browser promise, optional filesystem enhancements, and the exact evidence required before adding a desktop host or backend service.

TL;DR

The GitHub Pages PWA is the complete required product, not a demo. Portable Electron is an accepted optional host, specified through UPD-001, for project folders and direct exports. Both hosts use shared adapters and canonical data. Backends remain deferred until a funded product update requires one.

Baseline web contract

Every accepted product stage must remain complete on the baseline web platform unless a later ADR explicitly changes the product boundary. Baseline capabilities include:

  • Open the product without installation or an account.
  • Create and manage local projects in IndexedDB.
  • Install supported public packs from static files and use installed packs offline.
  • Compose, edit, preview, validate provenance, and recover work locally.
  • Export and import portable .spriteproject archives.
  • Download generic and engine-oriented output packages.
  • Continue installed workflows offline after initial caching.
  • Inspect storage, persistence, version, and update state.

Progressive filesystem access

Where supported, the product may offer a directory picker and retain a permission handle for user-approved export. This improves repeated engine delivery but does not become a required live link.

Shared host boundaries

interface ProjectRepository { /* list, load, save, snapshot, restore */ }
interface PackRepository { /* list, install, remove, read blobs */ }
interface ExportSink { /* download or write generated files */ }
interface PersistenceStatus { /* usage, quota, persistence, diagnostics */ }
interface HostCapabilities {
  directoryWrite: boolean
  continuousWatch: boolean
  processLaunch: boolean
  cloudSync: boolean
}

Domain and rendering modules consume project, pack, and export contracts. They do not branch on window.electron, browser name, filesystem paths, or backend presence. The UI presents capabilities from the host and always retains the documented fallback where one exists.

Desktop scope and future expansion triggers

ADR-002 accepts the portable desktop host for local file convenience. UPD-001 includes project folders, atomic Save/Save As, recent folders, portable archives, direct generic/Godot directory export, and safe unsaved close. Additional desktop capabilities still require a new formal update when an accepted user flow needs:

  • Continuous filesystem watching for a measured live-link workflow.
  • Launching or controlling a local engine, converter, or build process.
  • Managing project or pack data that repeatedly exceeds practical browser quotas.
  • Editing or watching arbitrary local content-pack folders beyond an approved project root.
  • Operating in controlled offline environments where browser installation and storage policy cannot be relied on.

Installer size, memory, packaging, signing, notarization, update hosting, security boundaries, and cross-platform validation become part of that decision. GitHub Releases may distribute binaries without a custom server, but signing and maintenance costs still require an explicit sustainable plan.

Backend adoption triggers

A backend is not justified by rendering, export, public pack discovery, or static documentation. It becomes a candidate only when an accepted promise requires one or more of:

  • Automatic cross-device project synchronization or account recovery.
  • Concurrent collaboration, comments, shared history, or team permissions.
  • Private pack access, purchases, subscriptions, or entitlement enforcement.
  • Marketplace submissions requiring moderation and transaction records.
  • Server-owned jobs that cannot run acceptably or securely on the user's device.

Before acceptance, a backend ADR must name funding, operator, data retention, privacy, authentication, abuse response, backup, incident recovery, shutdown/export strategy, and behavior when the service is unavailable.

Decision test

  1. Name the user promise

    Infrastructure cannot be proposed without a traceable scenario and task flow.

  2. Try the baseline

    Measure the GitHub Pages PWA flow with the real data size, browser, and downstream tool.

  3. Try progressive capability

    Use optional browser APIs only when a complete fallback remains viable.

  4. Prove the gap

    Capture objective failure or unacceptable friction rather than assuming native or cloud is better.

  5. Price the lifecycle

    Include build, signing, security, operations, migration, support, and shutdown, not only initial implementation.

  6. Write the next ADR

    Define shared adapters, data portability, fallbacks, and supersession of affected rules before implementation.

Default answer

When a requirement changes, run the Product Update Pipeline first. Improve the web baseline or desktop adapter according to the derived promise and host flows; do not add infrastructure from implementation preference alone.
