Architecture · Compatibility contract

Project File Compatibility

One versioned .spriteproject format that web and Electron can both open, edit, save, migrate, validate, and exchange without host conversion or semantic drift.

TL;DR

The same file works in both hosts. A .spriteproject file is the universal portable project format. The Electron project folder is its exploded canonical payload plus explicitly non-portable exports/ and .sprite-cache/ directories. Both hosts use identical schemas, canonical JSON, path rules, checksums, migrations, and pack locks. Host-only state never enters the file.

Compatibility is a product invariant

Web and Electron may differ in storage convenience, but they cannot create different project dialects. A file saved by either host must open and re-save in the other without an import conversion step, loss of supported data, changed rendering, changed attribution, or changed engine output.

Canonical representations

Portable file

project-name.spriteproject
  archive-manifest.json
  project.json
  packs.lock.json
  recipes/
  imported/
  provenance/
  README.txt

The interchange, backup, and cross-host file. Both web and Electron open and save it directly.

Electron project folder

project-name/
  archive-manifest.json
  project.json
  packs.lock.json
  recipes/
  imported/
  provenance/
  README.txt
  exports/
  .sprite-cache/

The canonical archive payload is present unchanged at the folder root. exports/ and .sprite-cache/ are host conveniences and are excluded when producing a portable file.

Version dimensions

Versions are independent. Hosts must not infer one from another.

VersionOwnsCompatibility behavior
archiveFormatVersionZIP entry contract, manifest, checksums, canonicalizationUnsupported future major fails before extraction or mutation.
projectSchemaVersionProject semantics, recipes, themes, preview, provenance referencesOlder supported versions migrate transactionally; future versions remain intact and unopened.
packLockVersionPack identity, version, checksum, source and embedding stateMissing exact packs block deterministic rendering but not safe archive inspection.
recipeSchemaVersionAsset selections, token overrides, producer-specific recipe fieldsMigrated independently so future producers do not force archive changes.
createdWithInformational product build identityNever used as a substitute for schema compatibility.

Archive manifest

{
  "archiveFormatVersion": 1,
  "projectSchemaVersion": 2,
  "packLockVersion": 1,
  "projectId": "uuid",
  "createdWith": "sprite-project/x.y.z",
  "entries": [
    {
      "path": "project.json",
      "mediaType": "application/json",
      "size": 1234,
      "sha256": "lowercase-hex"
    }
  ]
}
  • Entry inventory is complete and sorted by canonical path.
  • Checksums use lowercase hexadecimal.
  • The manifest does not checksum itself.
  • All listed entries must exist; unlisted entries fail validation except explicitly permitted non-semantic metadata in a later archive version.
  • Readers validate the entire manifest before committing project data.

Project schema evolution

Project schema version 2 is the only format written by UPD-001. It separates project metadata in project.json from the active character recipe in recipes/<recipe-id>.json. The project stores activeRecipeId, an ordered recipeIds array, exact pack-lock references, theme, preview, timestamps, and a non-negative integer revision. UPD-001 creates exactly one recipe; the shape reserves future growth without promising multi-recipe UI.

The verified schema-version-1 project migrates deterministically: preserve project and character IDs and all creative values, move the character value into one recipe document, resolve the exact pack lock, set revision zero, validate the complete graph, and require semantic, render, provenance, and export parity before commit.

Host-only data that must not enter the file

  • IndexedDB keys, object URLs, storage estimates, persistence grants, and browser permission handles
  • Absolute filesystem paths, drive letters, recent folders, file handles, inode/file IDs, and Electron IPC state
  • Window size, position, maximized state, theme preference, panel expansion, and development flags
  • Autosave timers, current write locks, transient errors, analytics identifiers, and temporary files
  • Generated thumbnails, render caches, service-worker caches, and .sprite-cache/
  • Engine export destinations and generated exports/ content

Folder and archive equivalence

  1. Read

    Electron validates the canonical root files exactly as the archive reader does. Folder access is not permission to accept a different schema.

  2. Save

    Electron canonicalizes JSON and updates manifest size/checksum entries before atomic replacement.

  3. Pack

    Electron creates .spriteproject from only manifest-listed canonical payload entries; outputs and caches are omitted.

  4. Unpack

    Opening a portable file as a folder writes canonical entries only after complete validation; existing folders require explicit conflict handling.

  5. Web import

    The web reader stores canonical content in IndexedDB without injecting browser state into the project document.

  6. Web save

    The web writer emits the same canonical payload and archive profile used by Electron.

Parity levels

Byte parity
Unchanged canonical payload entries have identical SHA-256 hashes after cross-host open and re-save. Container bytes may differ only where the archive profile explicitly permits.
Semantic parity
After excluding declared non-semantic metadata such as updatedAt, canonical project structures are equal.
Render parity
Identical project + exact packs produce identical frame and spritesheet hashes.
Provenance parity
Selected source, author, license, and credit records are identical and complete.
Export parity
Animation and Godot metadata are structurally identical; generated PNG hashes match.

Atomicity and recovery

  • Writers create a complete temporary file or folder payload, validate it, flush where available, and atomically replace the destination.
  • Never update project.json without updating its manifest size and checksum in the same logical save.
  • A failed save leaves the last valid file/folder intact and keeps the project visibly dirty.
  • Migration never overwrites the only source copy. Web keeps an IndexedDB snapshot; Electron keeps the source file/folder and writes a new temporary result.
  • Web optimistic concurrency uses an integer revision checked inside the IndexedDB transaction. Desktop saves compare SHA-256 fingerprints of every canonical entry read at open.
  • A revision or fingerprint mismatch writes nothing and requires explicit reload, overwrite with a recovery checkpoint, save-as/copy, or cancel resolution.

Implementation source of truth

This profile is binding for UPD-001. JSON Schemas, canonical serializers, archive readers/writers, folder adapters, migrations, and compatibility fixtures must be shared packages consumed by both hosts, not separately reimplemented behavior.
