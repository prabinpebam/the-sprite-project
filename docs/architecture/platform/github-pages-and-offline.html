GitHub Pages & Offline Delivery

How the product is built, hosted, updated, cached, and distributed with no required application server or paid runtime infrastructure.

TL;DR

GitHub Pages serves a static Vite PWA from the repository subpath. The service worker caches a versioned app shell and compact official packs so installed users can work offline. No secret or server-only operation exists in the bundle. Large optional packs may use GitHub Releases when repository delivery becomes unsuitable.

Build contract

  • Production output is static and contains no Node.js runtime dependency.
  • Vite uses a relative or repository-aware base path so assets resolve under /the-sprite-project/.
  • All application URLs, worker URLs, pack URLs, and icons are tested from the deployed subpath, not only /.
  • Client navigation uses static-host-compatible routing. A direct refresh may not depend on server rewrites GitHub Pages does not provide.
  • Builds are deterministic from the lockfile and fail on lint, type, unit, traceability, or production-bundle errors.
  • No environment value placed in the client bundle is treated as secret.

Offline promise

After one complete online load, the installed PWA must support every flow that does not inherently fetch a new pack or open an external source link:

  • Open, create, rename, edit, autosave, recover, export, import, and delete local projects.
  • Use all installed packs and bundled starter content.
  • Compose, recolor, preview animations, inspect stored provenance, and generate generic or Godot packages.
  • Read local help required to complete those flows.

Offline mode may block discovery or installation of a pack not already present. The UI must distinguish offline and unavailable from missing, invalid, or not licensed.

Update lifecycle

  1. Discover

    A new service worker installs beside the active version without interrupting the current session.

  2. Protect work

    Before activation, complete pending project transactions and create any migration checkpoint required.

  3. Inform

    Show the available version and whether reload is required; never replace an active editing session silently.

  4. Activate

    Reload only after user confirmation or at a safe startup boundary.

  5. Migrate

    Run transactional data migrations before opening projects; preserve the prior known-good state on failure.

  6. Clean

    Delete obsolete application caches only after the new shell starts successfully; do not delete project stores.

Pack distribution

Bundled packs
Small, essential, versioned packs marked bundled: true in the static pack index, deployed with the application, and precached in the offline baseline.
Static pack index
A public JSON document containing pack ID, version, size, checksum, compatibility, source, and license summary.
Optional pack files
Immutable, versioned archives fetched on demand and verified before installation.
Large pack escape hatch
Move binaries to public GitHub Releases while keeping the index and checksums static; do not introduce a custom server only for file hosting.
Unavailable network
Installed packs continue working; unavailable remote packs retain metadata and an explicit retry state.

Offline readiness is computed from exact project pack locks. A lock is ready only when its version and checksum resolve to a bundled, checksum-validated local installation, or archive-embedded record. The UI reports Ready offline only when every lock resolves locally and names unavailable packs without substituting a different version.

GitHub Pages is not an application backend

It cannot safely hold secrets, authenticate private content, process uploads, run jobs, coordinate collaboration, or provide a transactional user database. Any future promise requiring those capabilities needs a separate ADR and sustainable operating model.

Privacy and security

  • Project data and imported art are not uploaded by baseline workflows.
  • External source links are ordinary user-initiated navigation, not background data transfer.
  • Pack archives are untrusted input: validate checksums, entry paths, expansion size, manifest schema, and media formats.
  • Content Security Policy should restrict scripts and workers to shipped assets and disallow unexpected connections.
  • Telemetry is absent by default. Any later telemetry must be separately documented, disclosed, optional where appropriate, and server-funded.
  • Do not claim cloud backup, account recovery, or private hosting.

Operational limits

RiskResponse
Repository or Pages size pressureKeep the app shell compact, split optional packs, and move large immutable archives to Releases.
Pages bandwidth or policy pressureMeasure real use, reduce eager downloads, and revisit ADR-001 only when the free delivery model is demonstrably unsustainable.
Broken deploymentRetain a known-good build artifact and documented rollback commit; service-worker fallback keeps installed users productive.
Stale installed PWAShow version/update state and retain compatibility with older project schemas until migration succeeds.
Subpath routing failureValidate the actual Pages URL and every asset/worker request before release.
GitHub outageInstalled cached application and local projects continue; new installation and optional pack download wait.
