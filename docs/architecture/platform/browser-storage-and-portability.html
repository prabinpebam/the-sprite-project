Browser Storage & Project Portability

How local projects remain transactional, recoverable, inspectable, and portable without accounts or a server database.

TL;DR

IndexedDB is the canonical browser workspace. localStorage holds only preferences; optional OPFS holds rebuildable large blobs; rolling snapshots protect autosave; and a versioned .spriteproject archive lets users back up and move work independently of browser storage.

Browser storage is continuity, not custody

Persistence requests reduce eviction risk but cannot guarantee survival after site-data clearing, private browsing, profile loss, or device failure. The product must make project archive export visible and routine.

Cross-host file contract

The exact serialization and compatibility rules are defined by Project File Compatibility. Web and Electron must open and save the same .spriteproject format; neither host owns a private dialect.

Repository boundary

Product code consumes asynchronous repositories rather than browser globals. This keeps project behavior testable and leaves room for a future desktop repository without forking domain logic.

Save protocol

  1. Debounce edits

    Collect rapid UI changes, but expose an immediate unsaved state and never hide write failures.

  2. Validate

    Validate schema, stable references, required packs, and serializability before opening the write transaction.

  3. Check revision

    Compare the expected revision to prevent an older tab from silently overwriting newer work.

  4. Snapshot known-good state

    Retain the prior valid project before replacing the current record.

  5. Commit atomically

    Write project, revision, indexes, and snapshot metadata in one IndexedDB transaction.

  6. Verify and announce

    Read back the stored revision, update visible status, and retain actionable diagnostics on failure.

Autosave and snapshots

  • Autosave follows meaningful changes, not animation-frame or hover state.
  • Maintain a small rolling history by count and age; never let snapshots grow without a documented bound.
  • Create named checkpoints before schema migration, pack replacement, or destructive recovery.
  • Preview state may persist, but transient playback timers and derived pixels do not.
  • Generated output is reproducible and should not inflate project records.
  • Show Unsaved, Saving, Saved, and Save failed as distinct user-visible states.

Import and conflict handling

  • Inspect and validate an archive before changing IndexedDB.
  • Reject path traversal, duplicate entries, unsupported compression, invalid checksums, and unbounded archive expansion.
  • Run migrations against a temporary copy; commit only after complete validation.
  • If the project ID already exists, offer replace, import as copy, or cancel; never guess.
  • Report missing packs before import and distinguish downloadable official packs from embedded or unavailable content.
  • Preserve the original archive when migration fails and provide diagnostics without exposing internal browser paths.

Quota and persistence

Persistence request
Call navigator.storage.persist() after the user creates or imports meaningful work, and explain the result without promising immunity from deletion.
Usage estimate
Use navigator.storage.estimate() to show approximate use and quota in a storage-management view.
Warning thresholds
Warn before writes are likely to fail, prioritizing archive export and cache cleanup.
Eviction priority
Delete derived caches first, unused official pack blobs second, snapshots within policy third; never silently delete current projects or owned imports.
Private browsing
Detect or explain that storage may be temporary; encourage immediate archive export.

Migration rules

  • Project, pack, and archive versions are independent.
  • Migrations are ordered, deterministic, idempotent where practical, and covered by real old-version fixtures.
  • Never destructively rewrite the only valid copy; snapshot before migration.
  • Unsupported future versions fail closed with an explanation and preserve raw data.
  • Rendering after migration must reproduce the prior semantic result or record an explicit compatibility change.
