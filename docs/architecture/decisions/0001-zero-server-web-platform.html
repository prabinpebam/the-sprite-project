Architecture decision · ADR-001

Zero-Server Web Platform

Adopt GitHub Pages as the required delivery surface and keep project storage, rendering, packs, recovery, and export on the user's device without paid application infrastructure.

TL;DR

Accepted on 2026-07-19. The production baseline is a static PWA hosted on GitHub Pages. IndexedDB owns workspace data; localStorage owns only small preferences; portable project archives provide user custody; all generation and export run client-side. Filesystem access is progressive enhancement. Electron and backend services are deferred until objective task-flow evidence crosses defined revisit thresholds.

Context

The product needs an immediately accessible character and world-asset workspace, but the project owner cannot fund application servers, databases, object-storage infrastructure, or always-on compute. The existing product already composes, recolors, persists, and exports entirely in the browser. Long-term workflows may eventually benefit from direct filesystem access, engine watchers, accounts, or collaboration, but those capabilities do not justify making the present product dependent on infrastructure that cannot be operated sustainably.

Decision

  1. Ship a static PWA

    GitHub Pages serves versioned HTML, JavaScript, CSS, the web manifest, service worker, and compact official packs.

  2. Keep computation local

    Composition, recoloring, provenance evaluation, project migration, archive creation, and engine export execute in the browser.

  3. Use IndexedDB for workspace state

    Projects, recipes, snapshots, installed pack metadata, and binary records live in transactional browser storage.

  4. Limit localStorage

    Only small non-authoritative preferences and pointers use synchronous key/value storage.

  5. Guarantee portable custody

    Users can export and import a versioned .spriteproject archive that does not depend on one browser profile.

  6. Treat local filesystem access as optional

    Supporting browsers may write to a user-approved directory; download and archive flows remain the cross-browser contract.

  7. Defer infrastructure by evidence

    Desktop and backend hosts require a new ADR tied to flows that demonstrably cannot meet their promise inside this platform.

Consequences

Benefits

  • Zero required runtime hosting cost
  • No account or network dependency for creative work
  • Private-by-default project data
  • Immediate browser access and optional PWA installation
  • Static deployment with a small operational surface
  • Shared core remains reusable by future hosts

Costs and limits

  • Browser storage can be cleared or denied persistence
  • No automatic cross-device synchronization
  • No continuous engine filesystem watcher
  • Pack sizes must respect static-site and browser-storage limits
  • Updates must coordinate with open local projects
  • Users need visible backup and recovery guidance

Binding implementation rules

  • Domain modules must not import IndexedDB, localStorage, service-worker, download, or filesystem APIs.
  • All persistent data has an explicit schema version and migration path.
  • Autosave uses transactional writes and never overwrites the last known-good snapshot before verification.
  • The product requests persistent storage but behaves correctly when the browser denies it.
  • Users can inspect storage status, export a backup, import a backup, and recover from an interrupted save.
  • The application shell and bundled starter content work offline after the first successful load.
  • No production secret, private key, or runtime configuration requiring secrecy enters the GitHub Pages bundle.
  • No product text may imply cloud backup, cross-device sync, or server custody while those capabilities do not exist.

Revisit triggers

This ADR remains accepted until one of these conditions is demonstrated with task-flow evidence:

  • A required workflow needs continuous filesystem watching or process control that downloads and user-approved directory access cannot satisfy.
  • Verified pack or project sizes repeatedly exceed browser storage and static-delivery limits for the target audience.
  • Cross-device recovery, collaboration, entitlement, or private distribution becomes an accepted customer promise.
  • GitHub Pages terms or limits no longer support the static product responsibly.
  • Security, accessibility, or browser support prevents a required flow from meeting its objective oracle.
