Architecture decision · ADR-002

Dual-Host Local-First Product

Keep the complete zero-server GitHub Pages experience and add a portable Electron host for convenient project-folder and export-folder operations over the same product core.

TL;DR

Accepted on 2026-07-19; implementation pending through UPD-001. Web remains the required, zero-install, zero-server product. Portable Electron becomes an optional Windows x64 host for local file convenience and future desktop expansion. Project, pack, rendering, provenance, and export contracts remain host-neutral. Electron uses a sandboxed renderer and narrow typed preload API.

Context

The browser product provides immediate, free access and can manage ordinary work through IndexedDB, archives, downloads, and optional PWA installation. Some local workflows are materially more convenient with native file operations: opening a durable project folder, atomic Save and Save As, recent folders, and writing repeated generic or Godot exports directly into a selected directory. The product owner has accepted both distributions as part of the evolving MVP.

Decision

  • Maintain GitHub Pages/PWA as a complete required product with no Electron dependency.
  • Add a portable Electron Windows x64 distribution as an optional second host.
  • Reuse one React interface and TypeScript domain core, with host-specific repository and export adapters.
  • Keep canonical project, archive, pack, provenance, rendering, and export schemas independent of host.
  • Store Electron-only state such as recent folders and permission context outside the project document.
  • Use project folders for desktop persistence and .spriteproject archives for cross-host transfer.
  • Distribute portable ZIP artifacts through GitHub Releases; do not add an update server.
  • Defer live link, process launch, installers, auto-update, signing, macOS, and Linux to later formal updates.

Portable project folder

project-name/
  sprite-project.json
  packs.lock.json
  recipes/
  imported/
  provenance/
  exports/
  .sprite-cache/

The project folder and archive represent the same canonical semantics. Absolute paths, recent history, window geometry, permission handles, and host capabilities do not enter the canonical document. Caches are disposable; user-owned imports and provenance are portable.

Consequences

Benefits

  • Zero-install discovery remains available to everyone
  • Desktop users gain ordinary project-folder semantics
  • Repeated engine export avoids download and extraction friction
  • Future watchers and local tool integration have a safe host boundary
  • No application server or update server is introduced

Costs and risks

  • Two host adapters and packaging surfaces require regression coverage
  • Unsigned portable Windows builds may show SmartScreen warnings
  • Electron increases artifact size and security-maintenance responsibility
  • Cross-host migration and parity become permanent product obligations
  • Desktop convenience must not degrade the web baseline

Verification obligations

  • Run every new Electron flow through packaged UI automation, not a browser substitute.
  • Run affected verified creative flows on web and Electron.
  • Round-trip real projects web → Electron → web and Electron → web → Electron.
  • Assert semantic project, render, provenance, generic export, and Godot export parity.
  • Test migration from the verified localStorage MVP into IndexedDB and project-folder/archive formats.
  • Test invalid IPC, traversal, permission denial, partial write, external modification, and unsaved close.
  • Verify the portable artifact on a clean supported Windows environment without admin rights.

Revisit triggers

  • Portable distribution friction materially blocks adoption and a funded signing/installer plan exists.
  • Electron artifact size or memory becomes a measured product blocker.
  • macOS or Linux demand becomes an accepted promise with packaging and signing resources.
  • A proven workflow requires live watching or process control, triggering a new scoped update.
  • Security maintenance cannot be sustained, requiring another host strategy or withdrawal.
