Architecture · Decision system

Product Architecture

Durable technical decisions, current platform contracts, and explicit triggers for changing them as The Sprite Project grows from a browser MVP into a complete asset studio.

TL;DR

The product has two accepted local-first hosts over one core: a complete zero-server web app on GitHub Pages and an optional portable Electron app for local file convenience. The web host is verified today; Electron is specified through UPD-001 but not yet implemented. Accepted decisions live in immutable ADRs, while platform pages document the current implementation contract.

How this section scales

decisions/ records why a choice was made and is superseded rather than rewritten. platform/ records the current operational contract and may evolve as accepted ADRs change it. Product scope remains in the MVP and North Star; verified implementation remains in Execution.

Capability boundaries

Architecture principles

Zero required server
No customer promise may silently depend on paid compute, a database, an account service, or a private API.
Local by default
Creative data remains on the device unless a future, separately accepted decision introduces an explicit transfer.
Portable ownership
Browser persistence improves continuity; a versioned project archive gives the user durable custody.
One product core
Storage, filesystem, and download behavior sit behind adapters; rendering and project rules never fork by host.
Progressive capability
Chromium-only filesystem APIs may improve a workflow but cannot become a baseline requirement.
Formal product updates
Host and infrastructure work begins only from a complete promise-to-evidence update; ADRs cannot bypass product derivation.

Decision lifecycle

  1. Propose

    Name the constraint, alternatives, affected promises, and objective evidence needed.

  2. Accept or reject

    Assign an ADR ID, owner, date, status, consequences, and revisit triggers.

  3. Implement

    Update platform contracts and traceable delivery scope without rewriting the historical rationale.

  4. Verify

    Exercise affected task flows in the actual delivery environment, including offline and recovery behavior.

  5. Supersede

    Keep the prior ADR intact, link the replacement, and migrate contracts and user data deliberately.
