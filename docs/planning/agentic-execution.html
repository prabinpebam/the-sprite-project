Delivery · Autonomous operating plan

Agentic MVP Execution

A dependency-driven program for delivering the character-to-Godot MVP with agents owning research, implementation, validation, evidence, and recovery while human involvement remains limited to decisions that cannot responsibly be automated.

TL;DR

Agents execute eleven evidence-gated work packages from bootstrap through release validation. They may make reversible technical decisions, install project-local tools, and reject unsuitable assets without waiting. Human input is required only for the starter-content legal policy, observed user validation, credentials or signing, and final release approval. The machine-readable source of truth is agentic-execution-plan.json.

Execution complete

The project-specific delivery model superseded package granularity where implementation evidence required it. The MVP Delivery Tracker records the verified result: all scoped flows and behaviors pass, and the original CC0 demonstration packs remove the pending third-party legal gate.

Future execution starts with product derivation

The initial MVP execution is complete. New requests do not enter this work graph directly. They first pass the Product Update Pipeline, which derives promises, scenarios, user-can statements, task flows, IA/navigation/views/UI, objective behaviors, exclusions, ADRs, and a fresh work graph. UPD-001 is the current example.

Unattended default

When the owner is unavailable, continue through every unblocked package, record assumptions, prefer the most reversible compliant choice, and quarantine uncertain content. Never wait for a preference that has a safe default; never infer legal approval, publish externally, or weaken a validation gate.

Operating contract

Plan from dependencies
Run any package whose prerequisites and required inputs are satisfied; parallelize independent packages.
Prove claims with artifacts
A package completes only when its commands pass and its declared evidence exists. Prose or screenshots alone do not satisfy a gate.
Keep the core headless
Composition, recoloring, provenance, persistence, and export remain executable without the product UI.
Prefer reversible defaults
Use adapters, versioned schemas, local fixtures, and project-local tools so later decisions do not invalidate the core.
Escalate precisely
Ask one bounded question only when every compliant fallback is exhausted or the decision is legally or commercially irreversible.
Leave a resumable record
After every package, update the build record with commands, results, artifacts, decisions, open risks, and the next runnable package.

Parallel execution

  1. Bootstrap once

    WP-00 establishes commands, boundaries, evidence locations, and deterministic local tooling.

  2. Run discovery and contracts together

    WP-01 inventories real art while WP-02 drafts schemas from known requirements. Reconcile them before either gate closes.

  3. Prove the headless chain

    WP-03 and WP-04 establish rendering; WP-05 can progress independently from the same source inventory. WP-06 joins them.

  4. Prove engine delivery before polishing UI

    WP-07 validates the export contract. WP-08 may scaffold around stable domain APIs but cannot invent renderer or license behavior.

  5. Attack assumptions

    WP-09 uses a second pack to expose false abstractions. WP-10 validates the actual user and distribution claims.

Agent recovery rules

FailureRequired response
Candidate asset has unclear provenanceQuarantine it, retain the evidence trail, select the next compatible candidate, and continue.
Palette substitution damages pixelsAdd explicit ramp metadata or a mask for that asset. If still unsafe, mark it non-themeable rather than using fuzzy replacement.
Pack layouts differRepresent the difference in pack data if it is a real source property. Reject the asset if describing it would require arbitrary executable hooks.
Godot text resources are unstableFall back to PNG plus generated import script inside the adapter; do not move Godot concerns into the renderer.
A deterministic comparison failsStop downstream work, isolate nondeterministic inputs or encoders, add a regression fixture, and rerun the same gate.
A package is blocked by human approvalMark only that gate blocked and continue every package that can use quarantined fixtures or non-shipping outputs.
Scope pressure appearsApply the MVP inclusion rule. Record the request in deferred work; do not expand the active package.

What is needed from the owner

Required before shipping · Content policy

Accept or replace the proposed starter policy: ship only assets whose chosen license path is CC0 or OGA-BY and whose authorship/source chain is complete. External legal review is the appropriate escalation if broader licenses are required.

Required for validation · Human sessions

Provide access to three target developers, or act as one participant, for observed character-to-Godot sessions. Agents prepare the build, script, capture, and issue synthesis.

Required for distribution · Authority

Provide secrets directly to the relevant terminal when signing, store access, domain control, or publishing credentials are eventually required. Agents must never request secrets through chat.

Required at the final gate · Release approval

Review the evidence index, unresolved risks, content policy, and user-validation findings, then explicitly approve or reject public release.

No response behavior

Until these inputs arrive, agents may complete technical implementation with non-shipping fixtures, prepare the candidate starter set, automate Godot validation, and stage a release candidate locally. They must not claim legal clearance, fabricate user observations, expose credentials, publish, deploy, or push remote changes.

Phase 1 is complete when

WP-00 through WP-06 have produced one real, legally traceable recipe that composes idle and walk in four directions, applies semantic recoloring, saves and rebuilds deterministically, and emits PNG, animation metadata, and exact credits through a headless command. That evidence, rather than completion percentage, authorizes the product UI and release work.
