Token-Based Coherence

A palette-aligned token system that makes every generator - characters, terrain, structures - share one consistent look.

TL;DR

Assets don't reference raw colors; they reference tokens - named roles like skin.base, foliage.mid, wood.shadow - that resolve to specific entries in the shared palette's color ramps. Because the palette follows the LPC style guide (and its recolor scripts only recognize on-ramp colors), tokens guarantee coherence: swap a theme and every generator recolors together, correctly, by construction.

Grounding

The LPC Revised Style Guide mandates a fixed palette organized as ramps, enforced by automated recolor scripts - "any color not in the ramps will not be recognized." Tokens turn that hard constraint into a design advantage.

The problem tokens solve

If each asset bakes in literal RGB values, mixing sources produces clashing palettes and manual recolor is tedious and error-prone (a top pipeline pain point). Coherence has to be structural, not a cleanup step.

Raw colors (today)

  • Each pack ships its own literal colors
  • Mixing packs = clashing palettes
  • Recoloring is manual, per asset
  • Off-palette pixels silently break recolor scripts

Tokens (proposed)

  • Assets reference token roles, not RGB
  • All packs resolve through one palette
  • Recolor = rebind tokens, project-wide
  • Only on-ramp colors are ever produced

Three layers: palette → ramps → tokens

The system is a small, strict hierarchy borrowed from design-token practice and fitted to pixel-art ramps.

Palette
The full fixed set of allowed colors (the LPC palette). Nothing renders outside it.
Ramps
Ordered shade sequences within the palette - e.g. a green ramp from darkest to lightest for foliage.
Tokens
Named semantic roles that point at a ramp + position, e.g. foliage.shadow → green ramp, step 2.

Lighting tokens too

Shading is tokenized as well

The style guide fixes light direction (top-front-left) and a specific shadow color (#2a1722 at 120/256 alpha). Those become tokens like shadow.cast and shadow.dark, so generated and hand-edited art shade identically - coherence extends to light, not just hue.

User-facing capabilities

  • User can retheme an entire project by rebinding tokens (summer → autumn, day → dusk).
  • User can define custom themes (token → ramp mappings) and reuse them across projects.
  • User can trust that every generated and edited asset stays on-palette and recolor-script-compatible.
  • User can recolor one asset by overriding specific tokens without breaking the shared theme.
  • User can see which tokens an asset uses, and where a token is used across the project.

Why tokens are the keystone

Coherence is the difference between "a pile of assets" and "a world." Tokens make coherence automatic and palette-legal, and they are the shared language that lets three generators - plus the manual editor - all speak the same visual dialect.
