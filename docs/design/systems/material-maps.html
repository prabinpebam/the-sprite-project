Material & Lighting Maps

Every asset ships not just a color image but a coordinated set of maps - normal, specular, emissive, and more - so 2D dynamic lighting "just works" in the engine.

TL;DR

Modern 2D games light sprites dynamically, which needs more than a diffuse image. Each generator and the editor must produce a matched map set - at minimum a normal map, with specular/roughness, emissive, height, ambient occlusion, and light/mask maps as needed. Maps are token-aware, travel with the asset through composition, and export in each engine's expected slots (e.g. Godot normal/specular, Unity secondary textures).

Grounding

2D lighting is an explicit polish-stage need in the indie pipeline research, and a recurring source of developer questions (normal maps, hiding sprites outside light, shader-driven lit sprites). Producing maps automatically removes a whole class of manual work.

Why maps are a first-class requirement

A flat color sprite can't react to a torch, a day/night cycle, or a glowing rune. Engines composite a sprite with its maps at render time to fake depth and material response. If the tool only outputs diffuse art, users are back in a separate program hand-painting normal maps - exactly the fragmentation this project removes.

Diffuse only (today)

  • Sprites look flat under dynamic lights
  • Normal/specular maps hand-painted elsewhere
  • Maps drift out of sync with edits
  • Per-engine wiring done by hand

Coordinated map set (proposed)

  • Sprites respond to lights out of the box
  • Maps generated alongside the diffuse image
  • Edits regenerate/propagate to all maps
  • Exported into each engine's native slots

How maps are produced

Because assets are generated from known parts and token-tagged materials, most maps can be derived automatically rather than hand-painted - then refined in the editor.

  1. Author-time normals from parts

    Layers/parts carry authored or inferred normals; compositing a character or building composites their normals too, so the assembled asset has a correct normal map for free.

  2. Material-driven specular/emissive

    Palette tokens already know a region is metal, glass, or a glowing rune, so specular and emissive maps are generated from token material class.

  3. Derived height/AO

    Height and ambient occlusion are estimated from shading and part depth; used to improve normals and grounding.

  4. Consistent light model

    All derivation obeys the style guide's fixed light direction (top-front-left) so maps agree with the baked shading.

  5. Edit & override

    The editor can paint/tweak any map with onion-skinned alignment to the diffuse frame.

Tokens tie shading and maps together

The style guide fixes the light direction and shadow color, and tokens tag material class. That shared information is exactly what a normal/specular/emissive generator needs - so maps stay consistent with the baked art and with each other by construction, across every generator.

Maps across the pipeline

Every generator emits maps

Characters, terrain, and structures/props each output the map set appropriate to their materials (e.g. water needs specular + normal for ripples).

Maps travel through composition

In the shared asset model, maps are part of the asset, so animation frames, atlases, and scenes keep every map in lockstep with the diffuse.

Animated maps

Animated assets animate their maps too - a flickering lamp's emissive, water's moving normals - via the same per-frame animation path.

Editor authors maps

Users can view and hand-edit any map channel in-app, with a lit preview to check the result immediately.

Engine-ready export

Maps export into each engine's expected structure so lighting works on import, not after manual setup.

Naming & packing conventions

Export follows predictable conventions - paired atlases and channel packing (e.g. a suffix like _n/_s/_e, or packing roughness/AO/height into one texture's channels) - so both engine importers and the live link and CLI can wire them up automatically.

User-facing capabilities

  • User can generate a coordinated map set (normal, specular, emissive, +optional) with any asset.
  • User can preview the asset under a movable light to check how the maps read.
  • User can toggle which maps an asset produces (keep it lean for unlit games).
  • User can hand-edit any map channel in the in-app editor, aligned to the diffuse.
  • User can rely on maps staying in sync across frames, atlases, recolors, and scenes.
  • User can export maps into the target engine's native lighting slots via files, live link, or CLI.

Why this is a core requirement

Dynamic lighting is table stakes for a polished 2D game, and authoring maps by hand is a real, documented pain. Because the tool already knows each asset's parts, materials (via tokens), and light model, it is uniquely positioned to generate the whole map set automatically - a capability the standalone LPC generator and most asset packs simply don't offer.
