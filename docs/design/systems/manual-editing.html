Design · Systems

Manual Edit & Create

The pixel-level work people do in Aseprite today - brought inside the tool, made token-aware, and kept in the same project as everything they generate.

TL;DR

Generation gets users 90% of the way; the last 10% is hand work - a tweak here, a custom frame there, a brand new asset. Today that means exporting to Aseprite, editing, and re-importing, which breaks the pipeline and the token coherence. The proposal is a built-in, token-aware pixel editor: draw, animate, and create with the same palette/tokens the generators use, so custom art is a first-class, re-exportable project asset - not a fork.

Grounding

The pipeline research shows Aseprite is the de-facto editor (layers, onion-skinning, palettes, sheet export, CLI). This proposal absorbs that core workflow rather than sending users out to it.

Why edit in-app at all

Today: round-trip to Aseprite

  • Export generated asset out of the tool
  • Edit in a separate app, off the token system
  • Re-import, re-slice, re-wire
  • Custom colors drift off-palette; credits desync

Proposed: edit in place

  • Open any generated asset in the built-in editor
  • Draw with the same palette/tokens and grid
  • Stays in the project; re-exports through one pipeline
  • On-palette by default; credits stay attached

Editor capabilities (the Aseprite-equivalent core)

Drawing

Pencil, eraser, fill, shapes, selection, and transforms at true pixel scale, with a zoomed + actual-size split view.

Layers

Compose in layers that map to the generator's slot/z-order model, so edits round-trip cleanly.

Animation

Frames, timing, and onion-skinning; author new animations or extend a generated set.

Palette / tokens

Paint with token-bound swatches; the palette is the project's palette, so nothing goes off-ramp.

What makes it different from Aseprite: it's token-aware

The editor is not a generic canvas - it understands the project's tokens and rules, so hand work inherits the same coherence guarantees as generated content.

Token brushes
Swatches are tokens (skin.shadow, wood.base), so edits recolor with the theme.
Palette lock
Optional guardrail that keeps painting on-ramp, preserving recolor-script compatibility.
Style assists
Grid, perspective guides (1:2 / 4:3), and the fixed light direction + #2a1722 shadow available as helpers.
Slot awareness
Edits know which slot/animation/direction they belong to, so they re-integrate without manual slicing.

Two modes: edit and create

Edit a generated asset

Open a character layer, tile, or prop, tweak pixels or frames, and it updates in the project (and via live link, in the engine) - no export/import.

Create from scratch

Make a brand-new asset that becomes part of a pack - a custom hat, a unique tile, a one-off prop - fully token-bound and creditable.

Closing the loop with packs and CLI

Hand-made content becomes reusable

A created asset can be saved into a pack, so bespoke work re-enters the generator catalog. And because editing is in-project, the CLI can batch-apply edits or regenerate variants - the same automation story as generation.

User-facing capabilities

  • User can open any generated asset and edit its pixels or frames in-app.
  • User can create a new asset from scratch on the project grid.
  • User can draw with token-bound swatches and stay on-palette automatically.
  • User can author or extend animations with onion-skinning.
  • User can save custom art into a pack for reuse in the generators.
  • User can see edits reflected live in the engine via the live link.
  • User never has to leave the tool for routine pixel work.

Why this is essential, not optional

Without in-app editing, the product is "generation only" and users still live in Aseprite for the finishing touches - fragmenting the pipeline the whole project aims to unify. A token-aware editor keeps the last 10% of hand work inside the coherent system.
