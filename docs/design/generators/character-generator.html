Character Generator (Expanded)

The character generator is only the nucleus. This proposal grows it from a fixed body-and-clothes tool into an extensible, pack-driven character system.

TL;DR

Today's LPC generator is characters + clothing. But the wider LPC world already includes separately distributed packs - pirates, horses/mounts, animals, and more - that users assemble by hand. The proposal: make the catalog a first-class, extensible pack system so themed and non-humanoid content (mounts, animals, riders) plug in cleanly, all sharing the same layer, palette, and license model.

Grounding

See the research on the ULPC generator, its gaps, and the art packs Pirates and Horses.

The problem with a fixed catalog

Great character content exists outside the base generator and is merged in slowly or used manually. The Pirates pack and the Horses/Rework sets are perfect examples - clearly compatible, yet not first-class.

Today

  • One curated catalog; new packs merged upstream slowly
  • Humanoid-centric; mounts/animals awkward or absent
  • Themed sets (pirates) live as separate downloads
  • No user-supplied packs

Proposed

  • Catalog = a set of installable packs
  • Subject types: humanoid, mount, animal, creature
  • Themed packs are one-click add-ons
  • Users/community can register their own packs

A pack-based catalog

Every piece of character content - base bodies, clothing, the pirate wardrobe, a horse, a rider rig - is a pack: a bundle of layers with slots, palette bindings, animation coverage, and license metadata. The generator becomes a host that composes whatever packs are installed.

Base packs

Body types and core anatomy (male, female, teen, child, muscular, pregnant) plus heads, hair, and faces.

Wardrobe packs

Clothing, armor, and themed sets like Pirates (coats, hats, hooks, peg legs) as drop-in add-ons.

Mount & animal packs

Horses, unicorns, pegasus, farm animals - as ridable mounts or standalone creatures.

Subject types beyond the humanoid

The base generator assumes a humanoid skeleton. Expanding means modeling other subject types, each with its own slot map and animation set, while reusing the shared palette and export machinery.

The rider/mount challenge

Mounts are the hard case: a character on a horse needs correct draw order (draw part of the horse, then the rider, then an overlay of the horse's near legs) and per-direction masks. The Horses research documents exactly this technique - the pack system should encode those compositing rules as data, not one-off code.

What a pack contains

Slots
Which layer positions it fills (e.g. torso, hat, mount.saddle).
Layers & z-order
The images per animation/direction and where they sit in the draw stack.
Palette bindings
Which recolorable regions map to which palette tokens (see token coherence).
Animation coverage
Which animations/frames the pack supports, so the UI can show gaps.
Compositing rules
Masks and overlays for hard cases like riders on mounts.
License metadata
Per-asset license + author, feeding automatic credits.

User-facing capabilities

  • User can browse and enable/disable packs, including themed and community packs.
  • User can build humanoids, mounts, animals, and rider+mount pairings.
  • User can see per-animation coverage and which packs fill which slots.
  • User can recolor any pack's regions through the shared palette tokens.
  • User can register a custom pack (their own art) that behaves like a built-in one.
  • User gets correct, merged credits spanning every pack used.

Why this matters

Turning the catalog into packs is what lets the character generator keep pace with the community's output (pirates today, something new tomorrow) and extend past humanoids - without forking the tool each time. It is also the model the environment and structures generators reuse.
