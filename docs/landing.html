The Sprite Project

Project Knowledge Base

One living workspace that carries the project from research through design, planning, and execution - the ultimate tool for sprite and game-asset generation.

TL;DR

This knowledge base is organized by phase, not by document type, so it scales as the project grows. Today the Research phase is populated; Design, Planning, and Execution are scaffolded and ready to fill. Each phase lives in its own folder with its own overview and categorized pages.

How this knowledge base is organized

Content is grouped into four sequential phases. Each phase is a folder in the navigation; new pages drop into the relevant phase and category without disturbing the rest.

Phases

1 · Research Active

The market, the LPC ecosystem, existing tools, art sources, and indie-developer needs and problems. The evidence base for every later decision.

2 · Design Planned

Product vision, UX flows, the sprite/data model, style-system rules, and architecture decisions - turning research into a shape.

3 · Planning Planned

Roadmap, milestones, scope, and task breakdown - sequencing the build into shippable increments.

4 · Execution Planned

Implementation notes, decisions log, changelog, and progress - the record of actually building it.

Working conventions

Structure
One folder per phase (research/, design/, planning/, execution/); category subfolders inside each phase.
Entry point
Every phase has an overview.html that orients and links into its pages.
Navigation
The sidebar is generated from folder paths, so adding a page or category needs no code - just a manifest entry.
Authoring
Pages are built with the docs-presentation (Slate) skill: body fragments, catalog components, visual-first.

Adding content

Drop a new page into the right phase/category folder and add one line to docs-manifest.json with its path, title, and order. The navigation, search, and table of contents update automatically.
