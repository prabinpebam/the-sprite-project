Governance · Product change control

Product Governance

How customer promises become scoped, designed, implemented, validated, and recorded without silently rewriting verified product history.

TL;DR

Governance is exceptional overhead, not the default workflow. Use the fast path for routine engineering, Git operations, maintenance, bug fixes, and implementation already inside an approved contract. Use the formal Product Update Pipeline only for material changes to customer promise/scope, observable workflow or objective oracles, or compatibility/risk boundaries.

Do not overapply governance

A commit, push, refactor, typo fix, test addition, dependency update, artifact cleanup, development command, or bug fix that restores an existing expected behavior does not trigger governance. Apply only the engineering validation appropriate to that task and keep moving.

Choose the lightest lane

LaneUse whenExamplesRequired overhead
Fast pathNo material product-contract changeCommit/push, refactor, formatting, existing-oracle bug fix, tests, build tooling, implementation of approved scopeImplement and validate narrowly

Default decision

When uncertain, use the fast path unless completing the task would make the approved specification or its evidence materially false or incomplete.

Separation of concerns

Product update
Defines what becomes true for users, complete experience scope, exclusions, and evidence.
ADR
Defines why a technical or platform direction is accepted and what would supersede it.
Execution package
Defines dependency-ordered implementation work mapped to product trace IDs.
Evidence run
Records observed product behavior and promotes status only when objective gates pass.
Verified baseline
Preserves historical truth for regression, migration, and rollback.

Material-change policy

  • A materially new UI capability needs a task-flow step and expected behavior; implementation of an already specified control does not need a new update.
  • A materially new task-flow outcome needs a user-can statement and scenario; fixing the existing outcome does not.
  • A new customer outcome or target user needs a promise; internal technical work does not.
  • A material platform, compatibility, privacy, security, legal, or distribution change needs a product update and reconciled ADR.
  • Removal or changed meaning needs impact classification, migration, consequence, and authority.
  • No verified status from code completion, screenshots alone, unit tests alone, or prose.
  • No rewriting prior evidence to reduce the apparent scope of a new update.
