Governance · Required workflow

Product Update Pipeline

The mandatory derivation and evidence path for material changes to approved or verified product contracts, deliberately excluded from routine engineering work.

TL;DR

Run the materiality test first. Routine work takes the fast path. A qualifying product update moves through request → baseline → impact → customer promise → scenarios → user-can statements → task flows → IA/navigation/views/UI → objective behaviors → exclusions → ADRs → execution → actual-UI evidence → release. The traceability validator must pass before implementation. Verified status comes only from generated evidence. The previous verified baseline remains intact.

Governance has a cost

Baseline freezes, impact analysis, exhaustive derivation, ADR reconciliation, and evidence expansion are useful only when product truth changes. Applying them to routine work slows delivery and obscures consequential decisions. The default is direct execution with proportionate engineering validation.

Canonical automation

The repository skill .github/skills/agentic-product-delivery/ defines this workflow, its traceability model, validators, and evidence rules. Update artifacts live under delivery/updates/UPD-NNN-name/.

Materiality test

Create or revise a formal update only when the request materially changes one or more of:

  • Promise or scope: customer outcome, target user, included/excluded behavior, quality, or recovery commitment
  • Observable workflow: user capability, task-flow outcome or branch, required interface surface, or objective behavior oracle
  • Compatibility or risk boundary: supported host, canonical data, migration, privacy/legal posture, security, distribution, or removal of behavior

Use one discriminator: would completing the task make the approved product specification or its evidence materially false or incomplete? If no, use the fast path.

Fast path · No governance

Do the work directly, validate proportionately, and do not create or amend a product update for:

  • Git status, commit, push, branch, merge, tag, or release bookkeeping
  • Formatting, typo, wording, link, or generated-document maintenance without requirement changes
  • Refactors, dependency maintenance, build/test tooling, development servers, packaging reruns, or artifact cleanup
  • Bug fixes that restore an existing expected behavior
  • Tests or evidence added for an unchanged contract
  • Implementation already authorized by an approved update
  • Performance, accessibility, or visual fixes that satisfy rather than change an existing oracle

Pipeline stages

  1. 1 · Capture request

    Assign UPD-NNN. Preserve the owner's language, rationale, constraints, date, source, and decision authority. Do not begin with a feature list.

  2. 2 · Freeze baseline

    Record exact Git revision, evidence run, statuses, trace counts, accepted ADRs, and rollback surface. Mark it immutable.

  3. 3 · Analyze impact

    Classify every affected promise, flow, contract, exclusion, and implementation area as retained, changed, added, superseded, removed, or unaffected.

  4. 4 · Define customer promise

    State user, context, trigger, observable outcome, quality, recovery, constraints, and required evidence. Avoid solution language until the outcome is clear.

  5. 5 · Derive scenarios

    Cover first use, returning use, alternatives, failure, recovery, offline, migration, accessibility, and every declared host.

  6. 6 · Derive user-can statements

    Create atomic observable powers. Each scenario has capabilities; every capability serves a scenario.

  7. 7 · Derive task flows

    Define entry, preconditions, visible actions, system responses, branches, cancellation, recovery, outputs, and completion. Split by user intent, not screen.

  8. 8 · Design the experience

    Derive information architecture, navigation, views, dialogs, controls, status surfaces, responsive behavior, and accessibility. Remove interface with no flow.

  9. 9 · Define objective behavior

    Give every material step an EB-* oracle: visible state, focus, pixels, persistence, file bytes, downstream import, error, recovery, or absence of anomalies.

  10. 10 · Close scope

    Record every plausible adjacent expectation as included or an explicit OUT-* decision with consequence and disposition.

  11. 11 · Reconcile architecture

    Add or supersede ADRs, host adapters, security boundaries, data formats, migrations, deployment, and rollback rules. ADRs support product scope; they do not replace it.

  12. 12 · Validate and approve specification

    Run the traceability validator. Review request fidelity, impact, completeness, exclusions, architecture, work packages, and release arithmetic. Approval permits implementation, not verification.

  13. 13 · Execute by dependency

    Implement vertical user flows through work packages mapped to trace IDs. After each edit, run the narrowest falsifying check and corresponding actual-UI flow.

  14. 14 · Capture actual behavior

    Drive visible controls on real declared hosts. Capture raw UI, pixels, accessibility, files, migrations, IPC, console, network, filesystem, downstream tools, and anomalies.

  15. 15 · Verify and release

    Require 100% traceability, new and changed flows, behavior oracles, affected regression, migration, parity, and zero unresolved severity 0-2 issues. Preserve the new baseline.

Traceability gate

Promise coverage
Every CP has scenarios and transitive expected behaviors.
Scenario coverage
Every SC has one or more user-can capabilities.
Capability coverage
Every UC participates in one or more task flows.
Flow coverage
Every TF has steps, views, UI, navigation where applicable, and behavior oracles.
Interface necessity
Every view, control, and route serves a flow; unused interface is removed or excluded.
Scope closure
Every plausible expectation is included or deliberately excluded.
Execution mapping
Every work package delivers trace IDs and has an executable gate.

Multi-host rule

Every promise, scenario, capability, and flow declares web, desktop, cross-host, or shared. Host convenience may differ, but canonical project semantics, pack locks, rendering, provenance, and export contracts cannot diverge silently. Cross-host round trip and output parity are product behaviors, not only integration tests.

Change within an active update

  1. Amend request

    Record the new source and rationale in the change proposal or revision history.

  2. Re-run impact

    Identify newly affected and invalidated records, implementation, evidence, migration, and exclusions.

  3. Re-derive downstream

    Start from changed promises; never patch only the requested UI control or flow step.

  4. Invalidate status

    Changed and affected records return below verified until fresh evidence passes.

  5. Regenerate review pages

    Human-readable documentation remains generated from the canonical graph.

Formal review checklist

  • Does the update reproduce the owner request without adding hidden promises?
  • Is the baseline exact, immutable, and usable for rollback?
  • Did impact analysis include product, UX, data, architecture, security, operations, migration, and evidence?
  • Can every UI element be traced to a flow, capability, scenario, and promise?
  • Are failure, recovery, cancellation, accessibility, offline, and host boundaries explicit?
  • Are plausible adjacent expectations closed by scope or exclusion?
  • Do execution packages map to trace IDs and runnable gates?
  • Does release require fresh actual-UI evidence rather than implementation claims?
